WHY

Gateway API vs Ingress

Ingress has served us well, but it was designed when Kubernetes routing was simpler — it has no concept of roles, forces complex behaviour into annotations, and only speaks HTTP/HTTPS. Gateway API is the official successor — expressive, role-oriented, and protocol-aware.

FeatureIngressGateway API
Role separation (Infra / App / Route)✗ None✓ First-class
Protocol supportHTTP / HTTPS onlyHTTP, TCP, TLS, gRPC
Traffic splitting / canaryAnnotation hacksNative weights
Multi-namespace routingLimitedReferenceGrant
Header-based routingController-specificNative spec
Portability across controllersLow — annotation-dependentHigh — spec-driven
Gateway API is GA since Kubernetes 1.28. It's not experimental — it's the future of Kubernetes traffic management and where the entire ecosystem is heading.

STEP 03

Install Gateway API CRDs

k3s does not ship Gateway API CRDs. Install v1.5.1 — this is critical. Traefik v3.7+ requires exactly this version. Installing an older version (like v1.2.1) causes the BackendTLSPolicy not found error and GatewayClass stays Unknown permanently.

Version mismatch is the #1 failure point. Always match the CRD version to what your Traefik chart requires. The Helm install notes will always tell you the exact version needed.
bashinstall CRDs v1.5.1
# Standard channel — HTTPRoute, Gateway, GatewayClass, GRPCRoute
kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.5.1/standard-install.yaml

# Experimental channel — BackendTLSPolicy (required by Traefik v3.7+)
kubectl apply -f https://github.com/kubernetes-sigs/gateway-api/releases/download/v1.5.1/experimental-install.yaml
bashverify
kubectl get crd | grep gateway.networking.k8s.io
expected output

The critical port mapping

Traefik's internal HTTP entrypoint runs on port 8000, not port 80. Your Gateway listener must match this internal port — this is the most common mistake when starting out.

localhost:8080  → (k3d maps) →  k3d loadbalancer :80
↓
Traefik web entrypoint :8000  ← Gateway port: must match this
↓
HTTPRoute → Service → Pods

STEP 06

Gateway + HTTPRoute

The Gateway defines where traffic enters. The HTTPRoute defines where it goes. These are intentionally separate objects — in production, platform owns the Gateway, app teams own the HTTPRoutes.

yamlgateway.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: main-gateway
  namespace: demo
spec:
  gatewayClassName: traefik
  listeners:
  - name: http
    port: 8000  # Traefik internal entrypoint — NOT 80
    protocol: HTTP
    allowedRoutes:
      namespaces:
        from: Same
port: 8000, not 80. The Gateway listener port must match Traefik's internal web entrypoint address (--entryPoints.web.address=:8000), not the host-level port you use in curl.
yamlhttproute.yaml
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: whoami-route
  namespace: demo
spec:
  parentRefs:
  - name: main-gateway
    namespace: demo
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: whoami
      port: 80
bashapply + test
kubectl apply -f gateway.yaml

POC 01

Traffic Splitting / Canary

Weighted backend routing — progressively shift traffic from stable to canary without any annotation hacks.

localhost:8080
↓
HTTPRoute (weighted)
┌────────┴────────┐
whoami-v1 (80%)
whoami-v2 (20%)
STEP 1

Deploy whoami-v2

bash
kubectl apply -f - <<EOF
apiVersion: apps/v1
kind: Deployment
metadata:
  name: whoami-v2
  namespace: demo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: whoami
      version: v2
  template:
    metadata:
      labels:
        app: whoami
        version: v2
    spec:
      containers:
      - name: whoami
        image: traefik/whoami
        args: ["--name", "whoami-v2"]   # v2 identifies itself in response
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: whoami-v2
  namespace: demo
spec:
  selector:
    app: whoami
    version: v2
  ports:
  - port: 80
EOF
STEP 2

Update HTTPRoute with weighted backends

yaml
kubectl apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: whoami-route
  namespace: demo
spec:
  parentRefs:
  - name: main-gateway
    namespace: demo
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: whoami      # stable
      port: 80
      weight: 80
    - name: whoami-v2   # canary
      port: 80
      weight: 20
EOF
STEP 3

Verify the split

bashcount distribution
# Send 50 requests and count which version responds
for i in $(seq 1 50); do
  curl -s http://localhost:8080 | grep "Name:"
done | sort | uniq -c
expected output
40 Name: whoami
10 Name: whoami-v2
STEP 4

Simulate canary promotion

bashprogressive rollout
# Promote to 50/50
kubectl patch httproute whoami-route -n demo --type=json \
  -p='[
    {"op":"replace","path":"/spec/rules/0/backendRefs/0/weight","value":50},
    {"op":"replace","path":"/spec/rules/0/backendRefs/1/weight","value":50}
  ]'

# Full promotion — 100% to v2
kubectl patch httproute whoami-route -n demo --type=json \
  -p='[
    {"op":"replace","path":"/spec/rules/0/backendRefs/0/weight","value":0},
    {"op":"replace","path":"/spec/rules/0/backendRefs/1/weight","value":100}
  ]'
vs Ingress: With nginx Ingress you needed nginx.ingress.kubernetes.io/canary-weight: "20" — a nginx-specific annotation that breaks on any other controller. Gateway API weights are controller-agnostic spec.
POC 02

Path & Header Based Routing

Route traffic based on URL paths and HTTP headers — enabling feature flags, API versioning, and beta user targeting natively.

localhost:8080/v1/*                  →  whoami (stable)
localhost:8080/v2/*                  →  whoami-v2 (canary)
localhost:8080/api/*                  →  whoami (stable)
Header: X-User: beta                →  whoami-v2 (canary)
Header: X-Version: v2 + /api/*   →  whoami-v2 (canary)
STEP 1

Path-based routing

yamlpath routing
kubectl apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: HTTPRoute
metadata:
  name: whoami-route
  namespace: demo
spec:
  parentRefs:
  - name: main-gateway
    namespace: demo
  rules:
  - matches:
    - path:
        type: PathPrefix
        value: /v1
    backendRefs:
    - name: whoami
      port: 80

  - matches:
    - path:
        type: PathPrefix
        value: /v2
    backendRefs:
    - name: whoami-v2
      port: 80

  - matches:
    - path:
        type: PathPrefix
        value: /api
    backendRefs:
    - name: whoami
      port: 80

  - matches:
    - path:
        type: PathPrefix
        value: /
    backendRefs:
    - name: whoami
      port: 80
EOF
STEP 2

Test path routing

bash
curl -s http://localhost:8080/v1/health | grep "Name:"   # → whoami
curl -s http://localhost:8080/v2/health | grep "Name:"   # → whoami-v2
curl -s http://localhost:8080/api/users | grep "Name:"   # → whoami
curl -s http://localhost:8080/         | grep "Name:"   # → whoami
STEP 3

Header-based routing

yamlheader routing
kubectl apply -f - <<EOF

POC 04

Multi-Namespace Routing with ReferenceGrant

The most production-realistic pattern — platform team owns the Gateway, app teams own their routes. ReferenceGrant is the explicit security boundary between namespaces.

infra namespace
Gateway (main-gateway)
team-a namespace
HTTPRoute /app-a
Service: app-a
ReferenceGrant ✓
team-b namespace
HTTPRoute /app-b
Service: app-b
ReferenceGrant ✓
Without ReferenceGrant, cross-namespace references are blocked by default. This is a deliberate security boundary — an HTTPRoute in team-a cannot reference a Gateway in infra unless team-a explicitly grants it.
STEP 1

Create namespaces

bash
kubectl apply -f - <<EOF
apiVersion: v1
kind: Namespace
metadata:
  name: infra
  labels:
    team: platform
---
apiVersion: v1
kind: Namespace
metadata:
  name: team-a
  labels:
    team: alpha
---
apiVersion: v1
kind: Namespace
metadata:
  name: team-b
  labels:
    team: beta
EOF
STEP 2

Deploy Gateway in infra namespace

Each listener gets a unique hostname — required because both share port 8000 and HTTP protocol. The port + protocol + hostname combination must be unique per listener.

Common error: Combination of port, protocol and hostname must be unique for each listener. If two listeners share port + protocol, they must differ by hostname. You cannot use the same port + protocol twice without unique hostnames.
yaml
kubectl apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: main-gateway
  namespace: infra
spec:
  gatewayClassName: traefik
  listeners:
  - name: http
    port: 8000
    protocol: HTTP
    hostname: "team-a.local"      # unique hostname makes this valid
    allowedRoutes:
      namespaces:
        from: Selector
        selector:
          matchLabels:
            team: alpha
  - name: http-b
    port: 8000
    protocol: HTTP
    hostname: "team-b.local"      # different hostname = unique combo
    allowedRoutes:
      namespaces:
        from: Selector
        selector:
          matchLabels:
            team: beta
EOF

STEP 4

Create ReferenceGrants

Each team explicitly grants the Gateway in infra permission to reference their services. The ReferenceGrant lives in the target namespace — the team that owns the resource decides who can reference it.

yaml
kubectl apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: allow-infra-gateway
  namespace: team-a       # lives in TARGET namespace
spec:
  from:
  - group: gateway.networking.k8s.io
    kind: HTTPRoute
    namespace: team-a
  to:
  - group: ""
    kind: Service
---
apiVersion: gateway.networking.k8s.io/v1beta1
kind: ReferenceGrant
metadata:
  name: allow-infra-gateway
  namespace: team-b
spec:
  from:
  - group: gateway.networking.k8s.io
    kind: HTTPRoute
    namespace: team-b
  to:
  - group: ""
    kind: Service
EOF

kubectl get referencegrant -A
expected output

team-b allow-infra-gateway 5s

Demonstrate the security boundary

kubectl apply -f - <<EOF
apiVersion: gateway.networking.k8s.io/v1beta1

Troubleshooting

Common errors and fixes
GatewayClass ACCEPTED: Unknown
Cause: CRD version mismatch or Gateway provider not enabled. Install v1.5.1 CRDs (both standard + experimental) and ensure --providers.kubernetesgateway is in Traefik's args. Check: kubectl logs -n kube-system deploy/traefik | grep -i gateway
Failed to watch *v1.BackendTLSPolicy: the server could not find the requested resource
Cause: Only standard CRDs installed. Fix: Apply experimental-install.yaml from Gateway API v1.5.1.
Cannot find entryPoint for Gateway: no matching entryPoint for port 80
Cause: Gateway listener set to port 80 but Traefik's web entrypoint is :8000 internally. Fix: Change Gateway port: 80 to port: 8000.
Combination of port, protocol and hostname must be unique for each listener
Cause: Two listeners share the same port + protocol without unique hostnames. Fix: Add distinct hostname: fields to each listener (e.g. team-a.local and team-b.local).
HTTPRoute condition: RefNotPermitted
Cause: Missing or deleted ReferenceGrant in the target namespace. Fix: Create a ReferenceGrant in the namespace that owns the Service being referenced.
404 page not found
Cause: Route not attached to Gateway or Gateway not programmed. Check kubectl describe gateway -n <ns> for conditions. Ensure parentRefs in HTTPRoute matches the Gateway name, namespace, and sectionName.
